# 项目说明：多 Agent 广告运营自动化 MVP

## 1. 项目解决的核心痛点

很多广告运营团队每天都在重复做几件事：登录平台看数据、人工识别异常计划、决定暂停还是调预算、再把结果同步给团队。这个流程强依赖经验，也容易漏掉波动较快的计划。

这个项目希望解决三个问题：

1. 数据分散，人工巡检耗时
2. 运营动作依赖个人经验，不稳定
3. 复盘和留痕不完整，团队协作效率低

## 2. 我的方案

我把传统的人工巡检流程拆成多个 Agent 节点，每个节点只负责一段明确职责：

```text
创建任务
  -> CoordinatorAgent 分配流程
  -> DataCollectorAgent 拉取广告计划
  -> AnalysisAgent 识别异常与潜力计划
  -> StrategyAgent 生成运营动作建议
  -> ExecutionAgent 执行动作
  -> ReportAgent 输出总结与健康分
  -> NotifyAgent 同步飞书通知
```

这种设计的价值在于：

- 流程清晰，可追踪，可复盘
- 每个 Agent 都能单独替换或升级
- 容易接入大模型，不需要推倒重来

## 3. 当前版本已经完成的能力

- FastAPI 后端接口
- React 可视化前端
- SQLite 数据落库
- 多 Agent 编排逻辑
- 模拟广告平台执行层
- 飞书通知占位能力
- 运行日志和结果展示

## 4. 多 Agent 分工

| Agent | 职责 |
|---|---|
| CoordinatorAgent | 任务协调、状态流转、异常兜底 |
| DataCollectorAgent | 拉取广告计划数据 |
| AnalysisAgent | 判断高 CPA、低 CTR、高潜计划 |
| StrategyAgent | 生成暂停、降预算、扩量建议 |
| ExecutionAgent | 调用执行层完成动作 |
| ReportAgent | 生成结果总结与健康分 |
| NotifyAgent | 同步飞书并沉淀通知链路 |

## 5. 当前核心规则

- CPA > 80：暂停计划
- CPA > 60 或 CTR < 2.5%：标记为高风险
- 60 < CPA <= 80：预算下调 20%
- conversions >= 30 且 CPA < 30：预算上调 15%
- 风险计划越多，账户健康分越低

## 6. 为什么它适合 MiMo 100T 申请

它满足评估里最重要的几个点：

1. 有明确业务问题和可解释价值
2. 有完整 Agent 运行逻辑，不是空概念
3. 有代码、有界面、有日志，能直接上传 GitHub 证明
4. 后续能自然接入 MiMo 系列模型，继续增强决策能力

## 7. 可继续补强的方向

- 把规则判断升级成 LLM + 规则双层决策
- 接真实广告平台 API
- 增加审批流，避免高风险动作直接执行
- 加入任务队列和定时巡检
- 增加团队协作、权限、审计能力
